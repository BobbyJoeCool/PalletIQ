## Not screen-specific

### [Bug]: Alert sounds don't fire in production

- **Status:** Fixed in `[1.0.2]` — see `CHANGELOG.md`. Root cause:
  `staticwebapp.config.json`'s `navigationFallback.exclude` list didn't include `mp3`, so Azure
  Static Web Apps served `index.html` in place of the actual audio files in production.
- **Type:** Bug
- **Screen / Feature:** Not screen-specific (all `playAlert()` call sites — Error/Warning/Info tones)
- **Environment:** Production
- **Priority:** Major/Important

**Current / actual behavior:** Alert sounds (Error/Warning/Info tones via `playAlert()`) do not play in the live production environment.

**Desired / expected behavior:** Alert sounds should play correctly in production, matching their behavior in local development.

**Steps to reproduce:**

1. Open the app in production.
2. Trigger any alert (e.g. an error, an info message, a warning).
3. Observe no sound plays, despite the same action producing the correct tone in local dev.

**Why this matters:**

**Additional notes:** Verified working correctly in local dev (Playwright-driven browser check) when the real audio implementation shipped in `[1.0.0]`. Not yet root-caused in production — possible leads: `playAlert()`'s `.catch()` on `audio.play()` silently swallows any rejection (autoplay-policy block, asset 404, etc.), so the actual production failure mode isn't currently observable; worth checking whether the mp3 assets survive the production build/deploy the same way other static assets do, and whether the production origin's autoplay policy differs from local dev's.

### [Bug]: Alert sound plays with a delay after the error message appears

- **Status:** Fixed in `[1.0.3]` — see `CHANGELOG.md`. Root cause: `playAlert()` built a fresh
  `new Audio(src)` on every call with no preload, forcing a fetch/decode before playback could
  start; tones are now preloaded once and reused.
- **Type:** Bug
- **Screen / Feature:** Not screen-specific (all `playAlert()` call sites — Error/Warning/Info tones)
- **Priority:** Minor

**Current / actual behavior:** There is an approximate 1 second delay between an alert/error message appearing on screen and its corresponding sound playing via `playAlert()`.

**Desired / expected behavior:** The sound should play in sync with the message — no perceptible delay between the message appearing and the sound firing.

**Steps to reproduce:**

1. Trigger any alert (e.g. an error, an info message, a warning).
2. Observe the message appears immediately, but the sound plays roughly 1 second later.

## ELZ

### [Feature Change]: Predictable StorageCode-Size ordering in the zone summary panel

- **Status:** Fixed in `[1.0.3]` — see `CHANGELOG.md`. `/api/locations/empty-by-zone` now sorts
  each zone's breakdown ascending by size (`XS, HS, S, M, L`), matching the canonical size order
  already used elsewhere in the app.
- **Type:** Feature Change
- **Screen / Feature:** ELZ
- **Priority:** Minor

**Current / actual behavior:** In the per-zone summary panel, StorageCode-Size rows display in whatever order the API happens to return them, with no defined sort order.

**Desired / expected behavior:** StorageCode-Size rows should display in a predictable order — ascending by size (e.g. `CR-HS`, `CR-S`, `CR-M`, `CR-L`).

**Steps to reproduce:** Open ELZ, load a zone summary panel where a storage code has multiple sizes present, and observe the row order.

**Why this matters:**

**Additional notes:** `DevNotes/Screen-Specs/ELZ.md`'s Per-Zone Summary Panel section doesn't currently specify a sort order at all for its StorageCode-Size rows.
